# Design Context Template

本模板用于描述一个功能模块的设计上下文。AI 在基于 PRD 生成或扩展该模块时，应优先加载对应 Context，在现有产品基础上继续设计，而非从零开始。

---

## 元信息

```yaml
module: ""              # 模块标识（英文，如 task-management）
module_name: ""         # 模块中文名（如 任务管理）
version: ""             # 当前设计版本（如 v2.3）
last_updated: ""        # 最后更新日期（如 2026-04-20）
figma_source: ""        # Figma 设计稿链接（精确到 page 或 frame）
owner: ""               # 设计负责人
status: ""              # active | deprecated | planned
```

## 布局快照

描述该模块当前使用的页面布局结构。引用 `patterns/page-layout-patterns.md` 中的布局类型。

**必填项：**

- **布局类型**：布局 1（双栏）/ 布局 2（三栏）/ 布局 3（单栏）/ 布局 4（分栏）
- **页面基准**：1920×960
- **主要区域划分**：

```
用 ASCII 图描述当前模块的实际区域划分。
示例：

┌──────────────────────────────────────────────────┐
│  NavbarBar                        52px           │
├─────────┬─────────────────────┬──────────────────┤
│         │  Contenthead + Tabs │                  │
│  192px  │─────────────────────│  Contentsidebar  │
│         │  Toolbar            │  300px           │
│         │  Table              │  （选中行详情）   │
└─────────┴─────────────────────┴──────────────────┘
```

**特殊约定**（与标准布局的差异）：

- 例如：Content 区域内嵌 Tabs 切换，而非 Contenthead 中放置
- 例如：Contentsidebar 可收起，收起后 Content 占满

## 核心交互链路

按用户操作的先后顺序，描述该模块的主要交互流。每条链路标注触发方式和目标容器。

### 主链路

```
操作步骤编号. 用户动作 → 系统响应（目标容器）
```

示例：

```
1. 左侧树选择分类 → 表格数据刷新（Table）
2. 工具栏输入搜索关键词 → 表格筛选（Table）
3. 点击表格行 → 右侧边栏展示详情（Contentsidebar）
4. 侧边栏点击「编辑」 → 弹窗表单打开（Modal Dialog）
5. 表单填写并提交 → 弹窗关闭 + 表格刷新 + Toast 反馈
```

### 辅助链路（可选）

批量操作、导出、高级筛选等非主流程。

## 业务实体

描述该模块涉及的核心业务对象和字段。AI 生成页面时据此判断字段类型、排序和分组。

### 核心实体

```yaml
entity: ""              # 实体名称（如 Task）
description: ""         # 一句话说明
```

### 字段清单

| 字段 | 类型 | 必填 | 说明 | 对应组件 |
|------|------|------|------|---------|
| name | string | 是 | 任务名称 | Input |
| status | enum | 是 | draft / active / completed / archived | Select / Tag |
| assignee | user | 否 | 负责人 | UserPicker |
| priority | enum | 否 | low / medium / high / critical | Select / Tag |
| due_date | date | 否 | 截止日期 | DatePicker |
| description | richtext | 否 | 详细描述 | RichText |

### 字段分组（如果表单有分组）

| 分组名称 | 包含字段 | 分组规则 |
|---------|---------|---------|
| 基础信息 | name, status, assignee | 必填优先 |
| 计划信息 | priority, due_date | 时间相关 |
| 补充信息 | description | 非必填大字段 |

## 消费方式

### AI 生成新页面时

1. 读取本文件的「布局快照」和「Pattern 组合」，确定页面骨架
2. 读取「核心交互链路」，理解操作流的入口和目标容器
3. 读取「业务实体」，确定字段类型和对应组件
4. 读取「状态与空态」，补全非 happy-path 设计
5. 读取「设计决策记录」，避免回退已确认的决策

### AI 扩展已有模块时

1. 先读取本文件，理解模块现状
2. 对比 PRD 新增需求与现有结构的差异
3. 优先在现有 Pattern 组合中寻找承载位置
4. 仅当现有结构无法承载时，才调整布局或新增 Pattern
5. 变更后更新本文件的对应段落

### 维护方式

- **自动提取部分**：布局快照、Pattern 组合可通过 `miline-figma` 的 `design-context` 命令从 Figma 提取结构，再人工校对
- **人工维护部分**：业务实体、交互链路、设计决策记录、已知约束
- **更新时机**：每次设计稿大版本变更后同步更新
